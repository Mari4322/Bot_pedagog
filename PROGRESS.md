# Статус разработки бота «Педагог»

## Этап 1 — Выполнено ✅

### Структура и база данных
- Структура проекта (`teacher_bot/` с подпапками)
- БД SQLite: все таблицы (`users`, `children`, `hobbies`, `subscriptions`, `requests_log`, `settings`)
- Все запросы к БД (`queries.py`)

### Диалог пользователя
- Шаги 1–6: сбор данных (ребёнок, возраст, увлечение, тема, тревожность, резюме) с навигацией «Назад»
- Шаг 7: генерация ответа через polza.ai, счётчик `daily_count`, запись в `requests_log`
- Шаг 8: перегенерация — выбор нового хобби + комментарий, тема берётся из предыдущего запроса
- Шаг 9: новый запрос — полный сброс FSM и старт с шага 1

### Команды пользователя
- `/start` — приветствие (первый / повторный визит), переход к шагу 1
- `/help` — инструкция со списком команд и описанием шагов
- `/cabinet` — личный кабинет: тариф, лимиты, дата следующего платежа, статус
- `/pay` — заглушка (реализация в Этапе 2)
- `/cancel_sub` — заглушка (реализация в Этапе 2)
- Регистрация команд в меню Telegram через `set_my_commands()`

### Интеграция polza.ai
- Подключение через OpenAI SDK с подменённым `base_url`
- Обработка всех кодов ошибок по ТЗ:
  - `401` / `402` → лог + уведомление администратору
  - `408` → лог (таймаут)
  - `429` → retry через 5 сек (до 3 попыток)
  - `500` → лог + уведомление администратору
  - `502` / `503` → retry через 10 сек (до 3 попыток)
  - Все прочие → лог + уведомление администратору
- Пользователь при любой ошибке видит только вежливое извинение
- Счётчик `daily_count` не увеличивается при ошибках
- Автоматический fallback на `gpt-4o` при ошибке 400 (модель не найдена)

### Планировщик APScheduler (UTC+7)
- `00:00` — сброс `daily_count = 0` для всех пользователей
- `00:01` — деактивация подписок, у которых `next_payment_at < now` и `auto_renew = False`
- `00:02` — напоминание пользователям, у которых подписка истекает через ~1 день
- Каждый час — проверка баланса polza.ai с предупреждением администратору при низком балансе

### Панель администратора
- `/admin` — inline-меню с шестью кнопками
- `/change_model` — смена активной нейросети глобально (из списка `AVAILABLE_MODELS`)
- `/get_users` — выгрузка CSV всех пользователей
- `/get_children` — выгрузка CSV детей и увлечений
- `/get_logs` — выгрузка CSV всех запросов и ответов
- `/add_admin <tg_id>` — добавить администратора (команда и кнопка с FSM-вводом)
- `/delete_admin <tg_id>` — удалить администратора (с защитой от самоудаления)
- `/balance` — текущий баланс polza.ai с индикатором статуса

### Безопасность и защита
- Боты-аккаунты игнорируются на уровне `/start` (не регистрируются в БД)
- Боты-аккаунты заблокированы на уровне генерации (`_do_generate`)
- Проверка лимита запросов перед отправкой в нейросеть
- Защита от самоудаления у команды `/delete_admin`

## Этап 2 — Ожидает реализации ⏳

### 1. Оплата через ЮKassa — `/pay`
- Показывать тарифы кнопками (Базовый, Премиум с реальными ценами)
- Создавать платёж в ЮKassa → отправлять ссылку пользователю
- Принимать вебхук `payment.succeeded` через FastAPI-сервер
- По вебхуку обновлять `subscriptions`: тариф, `paid_at`, `next_payment_at`, `auto_renew = True`
- Подтверждать оплату сообщением пользователю

### 2. Отписка — `/cancel_sub`
- Показывать подтверждение с датой окончания доступа
- По подтверждению ставить `auto_renew = False`
- Объяснять, что доступ сохраняется до конца оплаченного периода

### 3. Смена тарифа
- Если подписка активна и пользователь нажимает `/pay` — предупреждать:
  «Новый период начнётся с сегодняшнего дня. Продолжить?»
- Кнопки «✅ Да, сменить» и «◀️ Назад»

### 4. Тарифы — заполнить цены и лимиты
- Согласовать и проставить реальные цены (`price`) и лимиты (`daily_limit`) для тарифов:
  - `basic` — Базовый
  - `premium` — Премиум

### 5. FastAPI-сервер для вебхуков ЮKassa
- Раскомментировать `fastapi` и `uvicorn` в `requirements.txt`
- Написать `webhook.py` с эндпоинтом `POST /yookassa/webhook`
- Запускать параллельно с ботом (asyncio или отдельный процесс)
- Заполнить `YOOKASSA_SHOP_ID`, `YOOKASSA_SECRET_KEY`, `WEBHOOK_URL` в `.env`
